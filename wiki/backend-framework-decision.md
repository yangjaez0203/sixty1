# 백엔드 프레임워크 기술 결정

- **날짜:** 2026-03-17
- **상태:** 확정
- **결정:** NestJS + Fastify

---

## 배경

sixty1은 실시간 동기화 스트리밍, 공유 플레이리스트, 미디어 처리(FFmpeg) 등 I/O 집약적인 기능을 요구한다.
백엔드 프레임워크 선택 시 다음 기준을 고려했다.

- **팀 생산성:** 학습 곡선, 코드 구조화 편의성
- **성능:** 동시 접속 처리, I/O 처리량
- **생태계:** 라이브러리 지원, 커뮤니티 규모
- **타입 안전성:** 런타임 오류 감소, 유지보수성

---

## 검토 후보

### 1. Go (Fiber v2)
**장점**
- 메모리 사용량이 낮고 처리 속도가 매우 빠름
- 단일 바이너리로 배포 간단

**단점**
- 팀 내 Go 숙련도 부족 → 학습 비용 높음
- 제네릭 기반 타입 시스템이 TypeScript 대비 유연성 낮음
- NestJS 수준의 구조화된 아키텍처 패턴 부재

**탈락 이유:** 팀 생산성과 학습 비용 문제

---

### 2. Ktor + Kotlin
**장점**
- Kotlin의 코루틴 기반 비동기 처리
- JVM 생태계 활용 가능

**단점**
- 실제 프로덕션에서 채택한 회사가 드물어 레퍼런스와 커뮤니티 자료가 부족
- 프론트엔드 담당자가 백엔드에 기여하려면 Kotlin + Ktor를 새로 학습해야 하는 비용이 과도함
- TypeScript를 공통 언어로 사용할 경우 대비 협업 효율 저하

**탈락 이유:** 낮은 업계 채택률과 팀 전체의 학습 비용 과다

---

### 3. Kotlin + Spring Boot
**장점**
- 엔터프라이즈 수준의 성숙한 생태계
- DI, AOP 등 검증된 아키텍처 패턴

**단점**
- JVM 특성상 기동 시 메모리 사용량이 높아 프리티어 EC2(t2.micro, RAM 1GB)에서 안정적인 운영이 어려움
- 애플리케이션 실행만으로 수백 MB를 점유해 다른 서비스(Nginx, Redis 등)와 함께 올리기 부담

**탈락 이유:** 프리티어 EC2 환경에서 운영 불가 수준의 리소스 요구

---

### 4. NestJS + Express
**장점**
- NestJS의 구조화된 모듈/컨트롤러/서비스 패턴
- Express의 넓은 미들웨어 생태계

**단점**
- Express는 단일 스레드 이벤트 루프 기반으로 Fastify 대비 처리량 낮음
- Fastify로 언제든지 교체 가능하므로 Express를 선택할 유인 없음

**탈락 이유:** 동일한 NestJS 환경에서 Fastify 대비 성능 열위

---

### 5. NestJS + Fastify ✅ (선택)
**장점**
- **구조화된 아키텍처:** 모듈/컨트롤러/서비스 레이어 분리로 팀 협업과 유지보수에 유리
- **TypeScript 네이티브:** 타입 안전성 확보, 런타임 오류 감소
- **고성능:** Fastify는 Express 대비 최대 2배 처리량, 실시간 스트리밍 요구사항에 적합
- **풍부한 생태계:** BullMQ, TypeORM/Prisma, Passport 등 필요한 라이브러리 모두 지원
- **낮은 학습 곡선:** 팀의 TypeScript/Node.js 경험 활용 가능

**선택 이유 요약**
팀 생산성(TypeScript), 성능(Fastify), 구조화(NestJS 아키텍처) 세 가지를 동시에 만족하는 유일한 후보였다.
